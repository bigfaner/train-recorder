# 1.4: Core Repositories (Plan, Day, Exercise, PlanExercise)

## Description

Implement the base repository class and the four core entity repositories: TrainingPlan, TrainingDay, Exercise, and PlanExercise.

## Reference Files

- design/tech-design.md — BaseRepository interface, Data Models Field Quick Reference
- design/er-diagram.md — TrainingPlan, TrainingDay, Exercise, PlanExercise entity details
- design/schema.sql — Table definitions for training_plans, training_days, exercises, plan_exercises

## Acceptance Criteria

- [ ] src/db/repositories/base.repository.ts implements BaseRepository<T> interface (findById, findByBizKey, findAll, create, update, deleteById)
- [ ] training-plan.repo.ts implements full CRUD for TrainingPlan with is_active toggle logic
- [ ] training-day.repo.ts implements CRUD with findByPlanBizKey for querying days of a plan
- [ ] exercise.repo.ts implements CRUD with soft-delete (is_deleted flag), findByCategory
- [ ] plan-exercise.repo.ts implements CRUD with findByTrainingDayBizKey, JSON parse/serialize for sets_config
- [ ] All repositories use snowflake biz_key for entity association (not auto-increment id)
- [ ] Integration tests with in-memory SQLite pass for all CRUD operations

## User Stories

- US-1 AC1-AC5: Plan creation, activation, switching (TrainingPlan repo)
- US-9 AC1-AC6: Exercise library management (Exercise repo with soft-delete, category filtering)

## Implementation Notes

- Base repository is generic over T; concrete repos extend with entity-specific queries
- Exercise soft-delete: update is_deleted=1 instead of DELETE; findAll filters is_deleted=0
- PlanExercise.sets_config stored as VARCHAR(2048) JSON string; parse on read, stringify on write
- TrainingPlan is_active toggle: when activating a plan, deactivate all others (single active plan constraint)
