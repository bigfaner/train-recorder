# 2.5: Unit Conversion + Data Export/Import Services

## Description

Implement three utility services: UnitConversion (kg/lbs display and storage), DataExport (JSON export of all training data), and DataImport (validate and merge imported JSON data).

## Reference Files

- design/tech-design.md — UnitConversion interface, DataExportService interface, DataImportService interface
- prd/prd-spec.md §5.11.1, §5.11.2 — Data export and unit switching specs

## Acceptance Criteria

- [ ] kgToLbs(kg) and lbsToKg(lbs) with conversion factor 2.20462
- [ ] displayWeight(kg, unit) formats weight with unit label
- [ ] storeWeight(input, unit) converts to kg for storage (always stores as kg)
- [ ] roundToPlate(kg) rounds to nearest achievable barbell plate combination
- [ ] exportData(range) generates JSON file with all entity arrays filtered by date range
- [ ] Export format: { training_plans, workout_sessions, ..., user_settings } arrays
- [ ] shareFile(filePath) triggers system share sheet (email/cloud/local)
- [ ] getEstimatedSize(range) returns approximate file size in KB
- [ ] validateFile(filePath) checks JSON schema, required fields, detects conflicts
- [ ] importData(filePath, conflictStrategy) merges data with conflict resolution (skip/overwrite)
- [ ] Import wrapped in transaction; rolled back on error
- [ ] Unit tests for conversion accuracy, export format, import validation

## User Stories

- US-14 AC1-AC3: Data export with range selection and share
- US-15 AC1-AC4: Unit switching with automatic conversion display

## Implementation Notes

- UnitConversion: internal storage always kg, display layer converts per user setting
- roundToPlate: standard barbell plates (1.25, 2.5, 5, 10, 15, 20, 25 kg); round to nearest achievable
- Export: use expo-file-system to write JSON; shareFile uses expo-sharing or React Native Share API
- Import: validate structure first (previewImport), then importData with transaction
- Conflict strategy: 'skip' ignores duplicate biz_keys, 'overwrite' replaces existing with imported
