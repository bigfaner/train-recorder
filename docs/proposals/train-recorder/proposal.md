# Proposal: Train Recorder — 渐进式力量训练助手

## Problem

力量举训练者缺乏一个围绕「渐进式加重」设计的训练记录工具。现有方案要么功能臃肿（面向泛健身用户），要么缺少关键能力（加重建议、进步分析），导致训练者只能依赖纸质记录或记忆来管理训练。

**具体痛点：**

1. **不知道该加多重** — 没有基于历史表现的加重参考，每次训练需要回忆上次做了什么、该不该加重量
2. **看不到进步趋势** — 无法直观查看某个动作的历史进步曲线，无法判断训练计划是否有效
3. **训练间隔混乱** — 没有系统化的训练安排和提醒，训练节奏不稳定
4. **记录过程繁琐** — 训练中记录数据不方便，打断训练节奏
5. **训练感受丢失** — 每次训练后的主观感受没有记录，无法关联分析训练质量和恢复状态

**为什么现在做：** 用户是力量举爱好者，采用周期化推/拉/蹲训练方式，每周训练 3-4 次，同时定期游泳。当前靠纸质和脑记，已积累数月但无法有效回顾和分析数据。随着训练水平提升，数据驱动的加重决策越来越重要。

**不做的代价：** 继续凭感觉训练，可能错过最佳加重时机、训练过量或不足、无法量化进步。

## Solution

做一个跨平台移动 App，围绕力量举周期化训练的完整闭环：

**训练前 — 智能建议：**

- 展示今日训练计划（基于推/拉/蹲分区）
- 根据上次训练表现，自动建议每个动作的目标重量
- 展示上次训练的完成情况和感受，供参考

**训练中 — 快速记录：**

- 按组记录重量和实际完成次数，一键操作
- 每组记录后自动启动组间休息倒计时（可自定义时长）
- 计时器到时提醒开始下一组

**训练后 — 回顾与感受：**

- 训练结束后记录主观感受（整体疲劳度、各动作感受）
- 自动汇总本次训练容量
- 标记本次训练是否完成目标，影响下次加重建议

**日常 — 分析与追踪：**

- 各动作的重量进步曲线图
- 训练容量趋势
- 身体数据变化趋势（体重、围度等）
- 其他运动（游泳等）的独立记录模块，支持自定义指标

### 加重建议规则

基于表现的自适应加重策略，每个动作可独立配置加重参数：

- 每个动作可设置独立的加重增量（如深蹲 5kg、卧推 2.5kg、二头弯举 1kg）
- 完成 → 下次建议加重（按该动作的配置增量）
- 未完成 → 下次保持相同重量
- 连续两次未完成 → 建议减重 10%

## Alternatives

### 方案 B：极简训练日志

只做记录和回顾，不做计划管理和加重建议。

**优点：** 开发量小，快速上线
**缺点：** 无法解决「不知道加多重」这个最核心的痛点，与其他健身 App 差异化不足

### 方案 C：继续使用现有方式

用纸质记录 + Excel 分析。

**优点：** 零开发成本
**缺点：** 无法在训练中实时获得加重建议，数据分析需要手动操作，记录容易遗漏

### 为什么选方案 A

四个核心痛点构成一个完整的训练闭环。方案 A 通过「计划 → 建议 → 记录 → 分析 → 再建议」的循环，同时解决所有痛点。方案 B 缺少最核心的加重建议，方案 C 不解决问题。

## Scope

### In Scope (MVP)

1. **训练计划管理**
   - 创建周期化训练计划（推/拉/蹲分区）
   - 定义每个训练日的动作列表和目标组数/次数
   - 支持多个计划模板的创建和切换

2. **动作库**
   - 内置常见力量举和辅助训练动作（深蹲、卧推、硬拉、推举、划船等）
   - 每个动作可独立配置加重增量（如深蹲 5kg、卧推 2.5kg）
   - 支持用户自定义动作，自定义动作同样支持加重配置
   - 从动作库快速选择动作编制训练计划

3. **训练执行与实时记录**
   - 选择当日训练开始执行
   - 逐组记录实际重量和完成次数
   - 组间休息自动倒计时，可自定义时长
   - 计时器到时振动/声音提醒

4. **渐进加重建议**
   - 根据上次完成情况自动建议本次目标重量
   - 完成 → 加重，未完成 → 保持，连续未完成 → 减重
   - 每个动作有独立的加重增量配置（在动作库中设置）

5. **训练日历**
   - 日历视图展示训练安排，快速查看哪天要训练
   - 每个训练日标注训练类型（推/拉/蹲/休息/其他运动）
   - 点击日期快速进入当日训练或查看历史记录

6. **训练后感受记录**
   - 训练结束后的整体感受评分（疲劳度、满意度）
   - 各动作的独立感受备注
   - 感受数据纳入加重建议的参考因素

7. **历史数据与进步分析**
   - 训练历史列表，按日期和训练类型筛选
   - 各动作的重量进步曲线图
   - 训练容量（总吨位）趋势
   - 个人记录（PR）追踪

8. **身体数据记录**
   - 记录体重、主要部位围度
   - 身体数据变化趋势图
   - 可关联训练周期查看

9. **其他运动记录**
   - 自定义运动类型（游泳、跑步等）
   - 自定义记录指标（距离、时间、配速、趟数等）
   - 独立于力量训练的历史记录和回顾

10. **数据存储**
    - 所有数据存储在设备本地
    - 数据导入/导出功能（后续版本可扩展云同步）

### Out of Scope (后续版本)

- 云同步 / 多设备同步
- 社交功能（分享、好友对比、排行榜）
- 动作教学视频
- 营养/饮食记录
- Apple Watch / Wear OS 配套应用
- 付费/订阅功能
- 教练端功能（给学员安排计划等）

## Constraints

- **跨平台：** 需同时支持 Android 和 iOS
- **离线优先：** 核心功能（记录、查看历史）必须离线可用
- **性能：** 训练中操作响应需在 200ms 内，不能打断训练节奏
- **单手操作：** 训练中可单手完成所有记录操作

## Risks

| 风险                                                 | 影响                               | 缓解措施                                                                   |
| ---------------------------------------------------- | ---------------------------------- | -------------------------------------------------------------------------- |
| 渐进加重建议规则过于简单，无法适应不同训练阶段的需求 | 用户对建议失去信任，回退到手动决策 | MVP 先用简单规则验证价值，后续根据使用反馈迭代；始终允许用户手动覆盖建议值 |
| 组间记录操作步骤过多，打断训练节奏                   | 用户放弃在 App 中记录              | 极简交互设计：记录一组数据不超过 2 次点击；利用上次数据预填充，减少输入    |
| 跨平台方案的性能或体验不如原生                       | 用户觉得「不够流畅」               | 关注关键路径的性能（训练记录、计时器），非关键路径可接受适度妥协           |
| 本地数据丢失（换设备、App 卸载）                     | 用户丢失所有训练历史               | 提供数据导出功能，引导定期备份；后续版本加入云同步                         |
| 功能范围过大导致 MVP 周期过长                        | 迟迟无法使用                       | 严格按优先级迭代：先做训练记录+加重建议核心闭环，其他功能依次加入          |

## Success Criteria

1. **核心闭环可用：** 能完成「创建计划 → 执行训练 → 记录数据 → 获得加重建议」的完整流程
2. **记录效率：** 记录一组数据不超过 2 次点击（含预填充）
3. **计时器可靠：** 组间计时器在后台切换 App 后仍能正确计时和提醒
4. **数据可视化：** 能查看任意动作的重量进步曲线和训练容量趋势
5. **跨平台一致：** Android 和 iOS 的核心体验一致
6. **离线可用：** 所有核心功能在无网络环境下正常使用
7. **自我使用验证：** 创作者本人连续使用 2 周以上，替代纸笔记方式
