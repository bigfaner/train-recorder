# 1.5: Workout Repositories (Session, Exercise, Set)

## Description

Implement repositories for the workout execution entities: WorkoutSession, WorkoutExercise, and WorkoutSet.

## Reference Files

- design/tech-design.md — Data Models Field Quick Reference
- design/er-diagram.md — WorkoutSession, WorkoutExercise, WorkoutSet entity details
- design/schema.sql — Table definitions for workout_sessions, workout_exercises, workout_sets

## Acceptance Criteria

- [ ] workout-session.repo.ts implements CRUD with findByDate, findByDateRange, findByStatus, findByTrainingType
- [ ] workout-exercise.repo.ts implements CRUD with findBySessionBizKey, findByExerciseBizKey
- [ ] workout-set.repo.ts implements CRUD with findByWorkoutExerciseBizKey
- [ ] WorkoutSet.create calculates is_target_met (actual_reps >= target_reps)
- [ ] WorkoutSession supports session_status state machine: in_progress → completed / completed_partial
- [ ] Integration tests with in-memory SQLite pass for all CRUD operations

## User Stories

- US-2 AC2-AC9: Set recording, mid-workout exit, background recovery (WorkoutSession + WorkoutSet repos)
- US-10 AC1-AC5: Mid-workout exit with partial completion (WorkoutSession status transition)
- US-13 AC1-AC4: Backlog workout recording (WorkoutSession is_backlog flag)

## Implementation Notes

- WorkoutSession.session_date is VARCHAR(10) 'YYYY-MM-DD' format for range queries
- WorkoutExercise.exercise_status enum: 'pending' | 'in_progress' | 'completed' | 'skipped'
- WorkoutSet.is_target_met computed on insert: actual_reps >= target_reps → 1, else 0
- No FK constraints — application-layer integrity via biz_key references
