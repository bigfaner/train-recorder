# Task Record: 2.2 Plan & Workout Repository Implementations

## Summary
Implemented TrainingPlanRepository and WorkoutRepository with SQLDelight backend. TrainingPlanRepository provides create/update/delete plans with nested training days and exercises, activate plan (deactivates all others), and application-level cascade deletes. WorkoutRepository provides full session lifecycle: create session with exercises, record sets, update exercise/session status, complete/partial complete, cascade delete, and backfill past sessions. Added SQLDelight queries for all plan/workout tables. Added new DomainError types (PlanNotFoundError, SessionNotFoundError, SessionLockedError, InvalidWeightError, InvalidRepsError, WorkoutExerciseNotFoundError).

### Key Decisions
- Application-level cascade deletes for all nested structures (no SQL FK) — delete order: set configs -> exercises -> days -> plan for plans, sets -> exercises -> session for workouts
- Session locking: completed and completed_partial sessions reject modifications (recordSet, updateExerciseStatus, updateSessionStatus throw SessionLockedError)
- recordSet automatically promotes exercise status from pending to in_progress on first set recorded
- backfillSession overrides isBackfill flag to true regardless of input, inserts full session with exercises and sets
- activatePlan deactivates all plans then activates target in a single transaction
- updatePlan deletes all old nested data then re-inserts new (full replace strategy)
